Redis数据结构

动态字符串SDS

redis虽然是C语言实现的，但是redis底层并没有使用C语言自带的字符串实现，而是自定义了结构体SDS，主要原因有以下几点：

  • 非二进制安全存储。C语言中的字符串本质是一个char数组并且以字符\0作为结束标志，如果要存\0的字符就会导致字符串内存截断。
  • 获取字符串长度时需要运算。即从头遍历知道出现\0。
  • 不可修改。

redis中有5个SDS，分别用来存储不同长度类型的字符串，从sdshdr5一直到sdshdr64，其中sdshdr8的源码为

__attribute__ ((__packed__))代表取消内存对其，这样做是为了更方便的进行指针的向前向后移动，因为redis实现的时候，是通过一个sds指针指向具体的sdshdr结构体，sds指针的起始位置指向sdshdr中buf数组的起始位置，而flag的低三位存储了当前结构体的类型，因此，指针向前读3为就可以得知当前是哪一种结构体，比如读到001，知道是sdshdr8，就知道len和alloc的长度为8位，再往前读就可以得到字符串长度等信息。

字符串扩容

当初始分配的字符数组容量满后，就要进行扩容，当新的字符串加进来，会获得一个新的长度newlen，也就是说扩容之后的长度至少要是 newlen，才能存下追加的字符串，但是 sds 一般会多留一些空闲空间，防止频繁扩容，那预留多少呢？这个判断就是来做这件事的，要是 newlen 小于 1M，就直接扩两倍；要是 newlen 已经超过 1M 了，就 1M、1M 地扩。

IntSet

当元素全都是整数的时候，可以使用intset来存储，这种存储更省空间。

intset 结构体

特点：

  • redis确保Intset的元素唯一、有序
  • 具备类型升级机制，可以节省内存
  • 底层采用二分查找来查询

新增元素

  • 首先要根据新增元素的编码值和encoding比较来判断是否要进行数组升级，也就是当前的编码值存不下新加入的值了；
  • 然后进行插入，通过二分查找找到插入下标，如过数组中已经存在该值，无需插入，否则将从当前位置到结尾的全部后移。
  • 如果发生了升级操作，要从后往前移动，防止覆盖已有值。

ZipList

压缩链表是在redis中元素存储不多的情况下使用的一种数据结构，占用连续的内存空间。

结构如下

  • zlbytes 值，它里面存了一个 int 值，占了 4 个字节，里面存的是整个 ziplist 占的总字节数；
  • zltail，int值，记录了最后一个 entry 在 ziplist 里面的偏移字节数，为了实现逆序遍历；
  • zllen，存了多少个元素；
  • zlend ，结束标志，始终为255；

对于每一个entry，长度不确定，结构为：

  • prevlen: 前一个元素的长度，占1个或者5个字节；
  • len：记录data的数据长度，占1,2或5个字节；
  • data：负责保存结点的数据

因此，如果知道某一个元素的起始地址，可以通过prelen+len+len中记录的data长度获取到下一个entry的起始地址，实现正序遍历；

同样，知道某一个元素的起始地址，减去prelen，就可以得到前一个元素的起始地址，实现逆序遍历；

连锁更新

头插一个新的entry导致后面的全部entry都要修改prevlen字段导致性能地下。

QuickList

ZipList虽然节省内存，但是申请内存必须是连续的空间，如果内存占用较多，申请内存效率低；

为了解决这个问题，需要限制ZipList的长度和entry的大小。

于是Redis3.2之后引入QuickList，是一个双端链表，不过每个链表的数据节点是一个个ZipList（Redis7之后全部用listpack替换zipList）

listpack

ZipList结果复杂会带来性能下降等问题，因此，redis7之后采用listpack替换ziplist，替换后的quickList长这样

listpack 说白了，跟 ziplist 差不多，也是一块连续的内存空间，但是呢，listpack 没有 ziplist 那么复杂的内部结构。

  • tot-bytes 占了 4 个字节，存了 listpack 占用的总字节数；
  • num-elements 占用 2 个字节，存了这个 listpack 里面的元素个数；
  • end-byte，它的值始终是 255。

对于每一个元素:

  • **encoding-type **存的是 element-data 部分的编码类型和长度。
  • data存放具体的数据；
  • backlen ，它存了当前这个 element 的总长度，这个总长度包含了的是 encoding-type 长度加上 data 部分的长度，不包含 backlen 自身的长度。

之所以有这个 backlen 存在，就和 ziplist 里面的 prevlen 一样，主要是为了支持从后向前遍历 listpack 。

如果知道一个元素的起始位置，向右读前一个element的backlen就能找到前一个数据；

Listpack的好处就是每个element是一个独立的个体，这样就不会发生连锁更新这种情况。

当发生hash冲突的时候，使用头插法解决冲突，这样插入效率高，而且因为redis单线程不会造成线程安全问题。

渐进式哈希

当负载因子大于等于1并且后台没有AOF和RDB时，或者负载因子大于5，会触发扩容；

当负载因子小于0.1时，会出发缩容(最小为4)；

扩容的时候，需要把旧的哈希表中的元素迁移到新的哈希表中，需要重新进行hash运算。(rehash的过程中需要把dict.reshashidx设置为>-1代表在rehash的进度)

rehash的过程不是一次性全部迁移完成的，在reshash的过程中，查询的时候需要对两个哈希表都进行查询，但是在插入操作的时候只需要插入到新的哈希表中即可。

每次对dict的操作都会检查是否存在rehash的状态，如果正在rehash，就进行部分的reshash，如此往复知道全部迁移完毕。

SkipList

跳表顾名思义就是跳着找， 与传统链表有几点差异：

  • 元素按照升序排序存储；
  • 节点之间包含多个指针，指针跨度不同；

跳表最多允许32级指针

数据类型具体用到的数据结构

String

基本编码方式是RAW，基于SDS实现，存储上限是512MB。

如果存储的SDS长度小于44字节，则会采用EMBSTR编码，此时对象头和SDS是一段连续的空间，申请内存是只需要调用一次内存分配函数，效率更高。

如果存储的字符串是整数值，则会采用INT编码，直接将数据保存在RedisObject的ptr指针的位置(刚好8字节)。

List

  • LinkedList:普通链表，可以双端访问，内存占用较高，内存碎片较多；
  • ZipList:压缩列表，可以双端访问，内存占用低，存储上限低；
  • QuickList：LinkedList + ZipList，可以从双端访问，内存占用较低，包含多个ZipList，存储上限高.

Redis3.2之后主要使用QuickList。

  • 为了效率和唯一性，set采用HT（HashTable）编码，Dict中的key用来存储元素，value统一为null。
  • 当存储的所有数据都是整数，并且元素数量不超过set-max-inset-entries时，set会采用IntSet编码，以节省内存。

Zset

Zset需要满足有序和唯一性，并且还要可以根据member找score，而skipList是有序的，但是无法满足唯一性以及根据member找score(因为跳表是根据分数查找的)，而hashTable能够满足唯一性，但是不满足有序性，因此Zset通过skipList + dict 实现。

但是这种存储方法非常的占用内存，当元素数量不多时，HT和SkipList的优势不明显，而且更消耗内存。因此zset还会采用ZipList结构来节省内存，不过需要满足两个条件：

  • 元素数量小于zset_max_ziplist_entrues，默认128
  • 每个元素都小于zset_max_ziplist_value字节，默认64

但是ziplist没法排序， 因此score和element是紧挨在一起的两个entry，element在前，score在后；

score越小越接近队首，score越大越接近队尾，按照score值升序排列。

Hash

  • 都是键值存储
  • 都需要根据键获取值
  • 键必须唯一

因此和Zset很像，区别就是去掉了skipList。

  • 当数据量较少的时候采用ZipList，用以节省内存，ZipList中相邻的两个entry分别保存filld和value
  • 当数据量较大时，hash结构会转为HT编码，也就是Dict，触发条件为：
    • ZipLsit中的元素个数超过hash-max-ziplist-entries(默认512)
    • ZipList中的任意entry大小超过hash-max-ziplist-value(默认64字节)

达到条件后变成dict

Redis网络模型

redis到底是多线程还是单线程

  • 如果仅仅是聊redis的核心业务部分(命令处理)，那就是单线程；

  • 如果是聊整个redis，就是多线程

    • 在redis版本迭代过程中，在两个重要的时间节点上引入了多线程的支持。
      • redis4.0，引入多线程异步处理一些耗时较长的任务，例如异步删除命令unlink
      • redis6.0，在核心网络模型中引入多线程，进一步提高对于多核CPU的利用率

那为什么redis这么久选择单线程

  • 因为redis是纯内存操作，执行速度非常快，因此性能瓶颈是网络延迟而不是执行速度，因此多线程并不会带来巨大的性能提升；
  • 多线程导致过多的上下文切换，带来不必要的开销；
  • 引入多线程面临线程安全问题，必然要引入线程锁这样的安全手段，实现复杂度变高，性能大打折扣。

Redis内存策略

过期策略

之前学习redis的时候知道，我们可以给redis的key设置一个存活时间（TTL），可以通过expire来实现，这其中有两个问题需要我们来思考：

  • redis是如何知道一个key是否过期，这背后的原理是什么?
  • 是不是TTL到期就立即删除呢？

RedisObject

前面说过，redis中所有的数据都是一个redisObject对象，该对象结构体为：

  • type 字段：占用 4 个 bit 位，指定了 robj 包装数据的具体类型，具体可选值有 OBJ_STRING、OBJ_LIST、OBJ_SET、OBJ_ZSET、OBJ_HASH，对应的是前面介绍的 Redis 底层数据结构。
  • encoding 字段：占用 4 个 bit 位，指定了 robj 包装数据的编码方式。对于相应的数据类型（即 type 值相同），因为存储的具体数据有所差异，对应的 encoding 编码方式也有所不同。
  • lru 字段：占用 24 个 bit 位，用来记录 LRU 的相对时间，在后面使用的时候会展开说。
  • refcount 字段：int 类型，用来记录当前 orbj 实例被引用的次数。有的时候，我们需要创建一个共享的 robj 对象，每新增一个使用方引用这个公共 robj 对象的时候，refcount 值就会加 1；每当引用减少一个的时候，refcount 值就会减 1；当 refcount 减到 0 的时候，就表示没人再使用这个 robj 对象了，也就可以释放这个公共 robj 对象的空间了。在效果上，这与我们 Java 里面的 GC 有点类似。
  • ptr 字段：void* 指针类型，用来执行 robj 底层包装的真正数据。

不管是键还是值，都是一个redisobject对象。

惰性删除

顾名思义，并不是在TTL到期后立即删除，而是在访问一个key时，检查TTL，如果过期就删除。

但是很明显这种方式如果一个key过期了但是一直不访问，就会造成内存的浪费。

周期删除

通过一个定时任务，周期性的抽样删除(注意不是全部遍历)。

  • LRU：最少最近使用。
  • LFU：最少频率使用。

那么redis如何实现LRU和LFU，也就是怎么记录频率等信息呢？

回忆一下redisObject对象，其中有一个lru字段，占用 24 个 bit 位，用来记录 LRU 或者LFU的相关信息。具体根据用户配置的不同策略决定。

如果是LRU，则以秒为单位记录最近一次访问时间，长度24bit；

如果是LFU，高16位以分钟为单位记录最近一次访问时间，低8位记录逻辑访问次数；

缓存穿透、击穿、雪崩

缓存穿透

客户端的请求在redis和数据库中都不存在，导致缓存永远不会生效，这些请求都会打到数据库上。

解决方案一般有三种：

  • 非发请求的限制。我们在接口处提前对请求数据的合法性进行校验，如果不合法就直接返回错误，避免进一步访问数据库和缓存。
  • 缓存空数据。当发生一次缓存穿透问题后，在缓存中存储一份查询数据的空记录，这样之后的请求就不会打到数据库。但是缺点也很明显，一是会消耗缓存的内存（给空值设置有效期），二是会造成短期的数据一致性问题。
  • 布隆过滤器。访问请求在查询缓存之前先查询布隆过滤器，如果布隆过滤器中没有这个数据，就直接返回，否则才放行。优点就是内存占用少，但是缺点是实现复杂，并且可能有误判，因为原理是通过hash来标记数据库中的数据，如果出现hash冲突就会导致误判，因此如果布隆过滤器显示没有这个数据，就是真的没有，但如果显示有，则不一定有，仍会有缓存穿透的风险。

缓存雪崩

造成缓存雪崩的原因主要有2个：

  • 大量热点key在同一时间失效导致查询全部打到数据库;
  • redis服务器宕机

解决方案：

  • 给不同的key插入随机TTL;
  • 利用redis集群提高服务的可用性；
  • 给缓存业务添加降级限流策略
  • 添加多级缓存

缓存击穿

缓存击穿也叫热点key问题，就是一个被高并发访问并且缓存重建业务比较复杂的key突然失效了，导致大量请求到达数据库。其实从定义上看缓存击穿像是缓存雪崩的子问题。

解决方法：

  • 互斥锁。当查询到缓存中没有数据时，先获取互斥锁，然后去数据库查询并且进行缓存重构，如果其他请求来了，没有查到缓存数据，然后锁也没拿到就挂起等待。这种方法的缺点就是互相等待，性能下降。

  • 逻辑过期。向redis中存储数据的时候设置永不过期，而是给数据中加一个逻辑过期字段，请求到达后比较逻辑过期字段，如果过期了，就新建一个线程并且尝试获取互斥锁去执行缓存重构，而自己直接先把旧数据返回。这个时候如果其他请求过来，同样查询到没有数据，就去获取锁，如果获取锁失败，则直接先把旧数据返回，而不是去等待。因此性能好，但缺点也显而易见：不保证一致性。有额外的内存消耗。
