jvm架构

Native关键字

凡是带了 native 关键字的，说明 java的作用范围达不到了，会去调用底层 C语言的库！
也就是说 native 方法 会进入本地方法栈。调用本地方法库 通过 JNI

JNI作用

扩展JAVA 的使用，融合不同的编程语言为Java所用。
最初是想融合 C 和 C++
JAVA 最开始的时候 C 和 C++ 市场很大，想要立足，必须要调用 C和C ++ 的程序。
JAVA 在内存区域中 专门 开辟了一块标记区域 本地方法栈【Native Method stack】，用来登记 Native 方法。
在最终执行的时候，加载本地方法库中的方法，通过JNI
目前 Native 方法用的比较少，除非是用Java开发硬件相关或者驱动本地的东西。比如说通过Java程序驱动打印机。
或者 Robot 机器人类，用到比较多，正常场景一般用不到。

现在通过Java调用其他接口实现方式很多，比如：socket 、webSocket、 http…

内存结构

内存结构示意图

程序计数器

作用

  • 记住下一条虚拟机指令的执行地址

特点

  • 线程私用
  • 不存在内存溢出
  • 如果线程正在执行的是一个Java方法，这个计数器记录的是正在执行的虚拟机字节码指令的地址；如果正在执行的是本地方法，则计时器的值应该为空。

虚拟机栈

  • 每个线程运行时所需的内存，称为虚拟机栈
  • 虚拟机栈是线程私有的
  • 每个方法执行的时候，Java虚拟机会同步创建一个栈帧，用于存放局部变量表，操作数栈、动态连接等信息。
  • 每一个方法被调用到结束，对应着栈帧的出栈和入栈。

本地方法栈

  • 线程私有，为虚拟机使用到的本地方法服务。

堆

  • 线程共享，堆中的对象都要考虑线程安全问题。
  • 通过new关键字创建对象都会使用堆内存。

堆内存溢出

堆内存虽然是JVM中内存中最大的一块，但也会出现内存溢出的情况：

  • 对象创建过多：当应用程序创建了大量的对象，并且这些对象无法被垃圾回收器及时清理时，堆内存可能会耗尽。这种情况下，可能需要调整堆内存的大小或者优化代码以减少对象的创建和保持时间。

  • 内存泄漏：如果应用程序中存在内存泄漏问题，即创建的对象在使用后无法被垃圾回收器释放，这些对象会逐渐占用堆内存空间，导致堆内存溢出。常见的内存泄漏情况包括意外的对象引用保持、缓存对象未被正确释放等。解决内存泄漏问题需要进行代码审查和内存分析，确保对象能够正确释放。

  • 过大的单个对象：如果应用程序中创建了一个巨大的对象，而堆内存的大小不足以容纳该对象，就会发生堆内存溢出。这通常发生在需要处理大量数据或者图像等情况下。解决这个问题的方法之一是增加堆内存的大小，但也需要评估是否有其他方式可以降低对象的大小或者改变算法以减少内存使用。

  • JVM参数配置不当：堆内存的大小可以通过JVM参数进行配置，例如使用-Xmx参数设置最大堆内存大小，如果参数配置不当，堆内存可能无法满足应用程序的需求，导致堆内存溢出。合理地配置JVM参数可以帮助避免这种情况的发生。

当堆内存溢出发生时，JVM将抛出java.lang.OutOfMemoryError异常。为了解决这个问题，可以增加堆内存的大小，优化代码以减少内存使用，检查和修复内存泄漏问题，或者使用一些内存管理工具来帮助定位和解决内存问题。

方法区

  • 线程共享
  • 用于存储类的元数据，包括类的结构信息、静态变量、常量池、方法字节码等。
  • 每个类在方法区只有一个唯一的实例，无论创建多少个该类的对象实例。

JDK8之前，类的元数据等主要存在永久代。永久代是方法区的一部分，用于存储类的元数据、静态变量、常量池等信息。

然而，由于永久代的大小固定且不够灵活，而且很容易发生内存溢出的问题，特别是在应用程序使用大量动态生成的类和字符串时。为了解决这个问题，从Java 8开始，永久代被元空间（Metaspace）所取代。

元空间（Metaspace）是JVM在Java 8及更高版本中引入的一种新的内存区域，用于存储类的元数据。与永久代不同，元空间的大小不再受固定限制，它可以根据应用程序的需要进行动态分配和释放。元空间的数据存储在本地内存（Native Memory）中，而不是JVM的堆内存中。

方法区内存溢出

如果类数量过多或者字符串常量过多等情况下，会发生方法区内存溢出等问题，JDK1.8之后使用直接内存存储，如果发生内存溢出，会报java.lang.OutOfMemoryError:Metaspace异常，如果使用JDK1.6，该版本JVM使用永久代，因此会报java.lang.OutOfMemoryError:PermGen space异常。

运行时常量池

运行时常量池（Runtime Constant Pool）是Java虚拟机在运行时存储常量的一块内存区域。它是方法区的一部分，用于存储字符串常量、类和接口的符号引用、字面量等。

运行时常量池的大小是在编译时确定的，它可以通过增加方法区的大小来扩展。在某些情况下，如果运行时常量池占用的内存过多，可能会导致方法区的内存溢出错误（例如”PermGen Space”或”Metaspace”溢出）。

字符串常量池(String Table)

在Java中，StringTable（字符串常量池）是一种特殊的数据结构，用于存储字符串常量。注意：该部分在JDK1.7之后被存放到了堆中，JDK1.6之前是在方法区的永久代中。

Java中的字符串常量池有以下几个特点：

  1. 字符串常量池是全局共享的：所有的字符串常量都存储在字符串常量池中，并且它们是全局可见和可共享的。这意味着多个相同的字符串常量实际上指向了同一个内存地址，以节省内存空间。
  2. 字符串常量池是在编译时创建的：在编译Java源代码时，所有的字符串常量会被编译器放入字符串常量池中。这意味着字符串常量的值在编译时确定，并在运行时保持不变。
  3. 使用字面量创建的字符串会被存储在字符串常量池中：当使用字面量方式创建字符串对象时（例如String str = "Hello";），如果字符串常量池中已经存在相同值的字符串常量，那么新的字符串对象将直接引用已存在的字符串常量。
  4. 使用new关键字创建的字符串会存储在堆中：当使用new关键字创建字符串对象时（例如String str = new String("Hello");），无论字符串常量池中是否已存在相同值的字符串常量，新的字符串对象都会在堆内存中创建。

需要注意的是，字符串常量池是一个重要的优化机制，它有助于减少内存占用和提高字符串比较的性能。然而，由于字符串常量池是全局共享的，需要谨慎使用，避免在字符串拼接和动态生成字符串等场景中产生大量无用的字符串常量，以免造成内存浪费。

总结：StringTable（字符串常量池）是Java中用于存储字符串常量的数据结构，它是全局共享的，并在编译时创建。使用字面量方式创建的字符串常量会存储在字符串常量池中，而使用new关键字创建的字符串会存储在堆中。这个优化机制有助于减少内存占用和提高性能。

StringTable性能调优

  • 调整 -XX:StringTableSize=桶个数。

    StringTable的底层是Hash表，如果Hash表桶的个数比较多，元素就会比较分散，哈希碰撞的概率就会变小，耗时也会变少。

  • 考虑使用String.intern()方法将字符串对象是否入池 。

直接内存

​ 直接内存不是JVM里的内存，而是操作系统里的内存。

  • 常见于NIO操作时，用于数据缓冲区（比如ByteBuffer使用的是直接内存）
  • 分配、回收成本较高，但读写性能高
  • 不受JVM内存回收管理

​ Java本身不具备磁盘的读写能力，他要想实现磁盘读写，必须调用操作系统提供的函数（即本地方法）。在这里CPU的状态改变从用户态（Java）切换到内核态（system）。

​ 当切换到内核态以后，他就可以由CPU的函数，去真正读取磁盘文件的内容，在内核状态时，读取内容后，他会在操作系统内存中划出一块儿缓冲区，其称之为系统缓冲区，磁盘的内容先读入到系统缓冲区中（分次进行读取），系统的缓冲区Java代码是不能够运行的，所以Java在堆内存中分配一块儿Java的缓冲区，即代码中的new byte[大小]，我们Java的代码要能访问到刚才读取的那个流中的数据，必须再从系统缓冲区的数据间接再读入到Java缓冲区，然后CPU的状态又切换到用户态了，然后再去调用Java的那个输出流的写入操作，就这样反复进行读写读写，把整个文件复制到目标位置。

在这里可以发现，由于有两块儿内存，两块儿缓冲区，即系统内存和Java堆内存都有缓冲区，那读取的时候必然涉及到这数据存两份，第一次先读到系统缓冲区还不行，因为Java代码访问不到他们，所以把系统缓冲区数据再读入到Java缓冲区中，这样就造成了一种不必要的数据的复制，效率因而不是很高。

当ByteBuffer把allocateDirect这个方法调用以后，意味着会在操作系统这边划出一块缓冲区，即direct memory，这段区域与之前不一样的地方在于这个操作系统划出来的内存Java代码可以直接访问，即系统可以同他，Java代码也可以访问他，即他对两对代码都是可以共享的一段内存区域，这就是直接内存。

即磁盘文件读到直接内存后，Java代码直接访问直接内存，比刚才的传统代码少了一次缓冲区里的复制操作，所以速度得到了成倍的提高。这也是直接内存带来的好处，他适合做文件的这种io操作。（更多的相关优化可以参考NIO更多相关知识）

直接内存分配和回收原理

  • 使用了 Unsafe对象完成直接内存的分配、回收，并且回收需要主动调用freeMemory方法
  • ByteBuffer的实现类内部，使用了Cleaner（虚引用）来检测ByteBuffer对象，一旦ByteBuffer对象被（Java）垃圾回收，那么就会由ReferenceHandler线程通过Cleaner的clean方法调用freeMemory来释放直接内存。
